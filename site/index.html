Testing the agent where it actually runs.

02 / INTRO

The question — and why it matters.

Qodo reports strong numbers on standard benchmarks. Benchmarks are clean. Real diffs are not. I wanted to see what happens when an IDE-resident agent meets a messy local diff with planted noise — and how it stacks up against the off-the-shelf /code-review commands a typical Claude Code user already has, and against community skills like code-review-skill.

The headline finding isn't recall. Two of the three tools caught every planted bug. The differentiator is calibration — knowing when not to fire.

03 / METHODOLOGY

How the test was run.

Dataset
browser-use pinned at commit 6b4ed253… with 10 surgical bugs injected by hand.
Bug Classes
Race conditions, command injections, resource leaks. Plus 1 trap: a gated eval() that's actually safe.
Ground Truth
Catch all 10 real bugs. Ignore the trap. Flagging the trap = False Positive.
Scoring
Tool names stripped. Sonnet 4.6 graded all 44 findings against a frozen rubric. Only Critical / High / Blocking counted.

06 / FUTURE WORK

Toward a production-grade benchmark.

This was a small-scale pilot. To turn it into something a team could actually rely on, these are the directions I'd push next:

Expand scope

5–10 repositories across Python, Go, TypeScript. 5 runs per track to absorb LLM variance.

Execution-based judging

Replace the LLM grader with an agent that writes & runs unit tests to prove a bug exists. Kills hallucinated grades.

Operational metrics

Latency (time-to-review) and cost (token spend). High F1 means little if review takes 4 minutes per diff.

Product & UX research

Do developers value precision (zero noise) or recall (catch everything)? When does the UI earn trust vs. burn it?

Adversarial trap suite

More gated-but-scary patterns — sandboxed exec, validated SQL, hashed user input. Stress-test FP rates.

Context testing

Diff-only vs. full-repo context. Does access to surrounding code change the verdict?
